最近折腾了一点RxJava，用过RxJava的同学肯定是知道subscribeOn和observeOn这两个方法的，这两个方法的作用都是调度线程，但是两个方法到底有什么异同什么时候用哪个……网上真的是众说纷纭……不得已之下只好自己瞎几把乱试也算搞明白了一点/doge

RxJava

A library for composing asynchronous and event-based programs using observable sequences for the JVM

通俗的说，就是一个在JVM上使用可观测的序列来组成异步的，基于事件的程序的库/(ㄒoㄒ)/~~

RxJava最主要的优势就在于他的简洁性，它使用了函数式响应式的编程风格，内置了很多方便好用的操作符，运用观察者模式使得在程序逻辑越来越复杂的情况下代码结构依旧能够保持简洁。

subscribeOn和observeOn

RxJava有一个很大的好处，就是能够很方便的在不同的线程中进行切换，切换线程的动作就是用subscribeOn和observeOn两个方法来实现的，经过我的瞎几把折腾，在下面整理一下这两个操作符的异同和效果……

  • observeOn

    specify the Scheduler on which an observer will observe this Observable

    指定一个观察者在哪个调度器上观察这个Observable

  • subscribeOn

    specify the Scheduler on which an Observable will operate

    指定Observable自身在哪个调度器上执行

是不是超级晦涩噢…………

经过试验，我总结了以下这么几点

  1. subscribeOn操作符指定整个调用链从最开始直到调用observeOn操作符在哪个线程执行

    e.g.

    语句1,2都是运行在io线程中，而语句3则运行在computation线程中。

  2. observeOn操作符可以多次调用，每次调用都会切换线程

    e.g.

    语句1会在当前线程执行，而2会在io线程，3会在computation线程分别执行。

  3. subscribeOn操作符只能调用一次，只有第一次会生效

    e.g.

    语句1,2因为调用了subscribeOn所以在io线程执行，语句3则会在computation线程执行，因为subscribeOn操作符只能调用一次，第二次不生效，所以语句4也在computation线程执行。

Conclusion

总结一下……RxJava一般的用法就是配合Retrofit之类的轮子从网络上，数据库中读取数据，在io或者其他线程中执行某些耗时操作，再切换回主线程对UI进行更新，然而这只是最初级的用法，这个玩意能做的还有很多很多，我还在不断的学习中嘻嘻
